# Pure-Go LaTeX backend for PDF reports

- **Status:** Accepted (NO-GO)
- **Layers:** Cross-cutting (reporting infrastructure, deployment image)
- **Related:**
- **Canonical:** [pdf-reporting.md](../platform/operations/pdf-reporting.md)

- [PDF generation migration to Go](pdf-go-chart-migration-plan.md)
- [Precompiled LaTeX plan](pdf-latex-precompiled-format-plan.md) (D-08)
- [Distribution packaging plan](deploy-distribution-packaging-plan.md) (D-09)

**Goal:** Evaluate whether `star-tex.org/x/tex` (a pure-Go TeX engine) can replace the
external `xelatex` binary in the proposed Go report pipeline (as outlined in
`../platform/operations/pdf-reporting.md`), eliminating the ~800 MB vendored
TeX Live tree shipped on the Raspberry Pi image and producing a single
self-contained Go binary that compiles `.tex` to `.pdf` in-process.

## TL;DR — Verdict

**NO-GO at present.** `star-tex.org/x/tex` is a faithful pure-Go translation of
Knuth's original `tex.web` (plain TeX, circa TeX78/TeX82). It is **not a LaTeX
engine, not a XeTeX engine, and ships none of the LaTeX2e macro infrastructure**
the templates require. The GitHub mirror is **archived (read-only since
2025-03-04)**; the SourceHut origin is at **v0.7.0** and explicitly pre-1.0.

The current templates depend on `fontspec` + `\setmainfont{AtkinsonHyperlegible}`,
which is **strictly XeTeX/LuaTeX territory** — it cannot run on a plain TeX
engine no matter how much macro support is bolted on, because it requires
engine-level primitives (`\XeTeXglyph`, OpenType layout tables, Unicode input
processing) that `star-tex` does not implement and is not architected to
implement.

The honest path forward is to **keep `xelatex` as the typesetter** and pursue
the existing D-08 precompiled-format plan to shrink the vendored TeX tree.
This document records the analysis so the question does not have to be
re-asked.

## What `star-tex.org/x/tex` actually is

`star-tex` was generated by running `web2go` against Knuth's canonical
`tex.web` source from TeX Live. The result is a pure-Go reimplementation of
the TeX78/TeX82 engine — the same engine that ships as `tex` (not `latex`,
not `pdftex`, not `xetex`) in any TeX distribution.

| Property                                   | Value                                                                                       |
| ------------------------------------------ | ------------------------------------------------------------------------------------------- |
| Engine family                              | Knuth plain TeX (not e-TeX, not pdfTeX, not XeTeX, not LuaTeX)                              |
| Output formats                             | DVI directly; PDF via internal `dvi/dvipdf` renderer                                        |
| Format files preloaded                     | None documented; plain TeX macros only                                                      |
| Unicode input                              | No — 8-bit input following classical TeX                                                    |
| Font formats                               | TFM (metrics), PK (bitmap), AFM (Type 1 metrics)                                            |
| OpenType / TrueType                        | Not supported as engine primitives                                                          |
| File search                                | `kpath` package, loosely mimicking kpathsea over a TDS tree                                 |
| Concurrency                                | Single-threaded; web2go faithful translation                                                |
| Status (SourceHut)                         | v0.7.0, pre-1.0, last release 2025-04-12                                                    |
| Status (GitHub mirror `go-latex/star-tex`) | Archived read-only 2025-03-04                                                               |
| Stated motivation                          | Originally to render LaTeX equations into `gonum/plot`; expanded to a full plain-TeX engine |

Consequence: anything that runs on `star-tex` must be a document the original
1982 `tex` binary could compile. That excludes the entire LaTeX2e ecosystem
unless `latex.ltx` and the macro-based half of every required package are
loaded at runtime, AND the engine supports the primitives those packages
actually call.

## Gap analysis: what the templates need vs what `star-tex` provides

In the proposed Go migration, templates would live in
`internal/report/tex/templates/`; today, the active templates are under
`tools/pdf-generator/pdf_generator/templates/`. The packages currently loaded
in the preamble are listed below. "Class" indicates whether the
feature is a LaTeX2e macro package (potentially loadable on top of a plain-TeX
engine) or an engine-level extension (requires e-TeX, pdfTeX, or XeTeX
primitives).

| Feature / package                                     | Class                    | Engine requirement                                      | `star-tex` supports?                           | Notes                                                                                                                                                                                                 |
| ----------------------------------------------------- | ------------------------ | ------------------------------------------------------- | ---------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| LaTeX2e format (`latex.ltx`)                          | Format                   | Plain TeX + e-TeX primitives                            | **No**                                         | LaTeX2e has required e-TeX (`\protected`, `\unexpanded`, etc.) since 2017. `star-tex` is plain TeX only.                                                                                              |
| `fontspec` + `\setmainfont`                           | Engine extension         | XeTeX or LuaTeX only                                    | **No**                                         | Requires `\XeTeXglyph`, OpenType layout, Unicode input. Hard wall.                                                                                                                                    |
| Atkinson Hyperlegible (TrueType)                      | Font                     | XeTeX/LuaTeX font loader                                | **No**                                         | Plain TeX consumes TFM + PK only. The font would have to be re-generated as Type 1 + TFM.                                                                                                             |
| `geometry`                                            | Macro package            | LaTeX2e + e-TeX                                         | **No** (no LaTeX)                              | Macro-only in principle; impossible without LaTeX2e format.                                                                                                                                           |
| `graphicx` + `\includegraphics{chart.pdf}`            | Engine extension         | pdfTeX, XeTeX, or LuaTeX `\pdfximage` / `\XeTeXpdffile` | **No**                                         | Plain TeX has no notion of embedded PDFs. The current pipeline already converts SVG → PDF via `rsvg-convert`; `star-tex` cannot place PDF content into output. At best it could place EPS via dvipdf. |
| `tabularx`, `supertabular`, `array`                   | Macro package            | LaTeX2e                                                 | **No**                                         | Macro-only, but require LaTeX format.                                                                                                                                                                 |
| `xcolor`, `colortbl`                                  | Macro + driver           | LaTeX2e + driver-level colour specials                  | **No**                                         | DVI specials route exists, but the LaTeX format is missing.                                                                                                                                           |
| `fancyhdr`                                            | Macro package            | LaTeX2e                                                 | **No**                                         | LaTeX format missing.                                                                                                                                                                                 |
| `hyperref`                                            | Macro + engine extension | LaTeX2e + pdfTeX/XeTeX/LuaTeX PDF primitives            | **No**                                         | Requires `\pdfobj`, `\pdfannot` (pdfTeX) or XeTeX equivalents. `star-tex`'s `dvi/dvipdf` is a back-end DVI-to-PDF renderer, not a PDF object writer accessible from macro layer.                      |
| `titlesec`, `caption`, `multicol`, `float`, `amsmath` | Macro package            | LaTeX2e + e-TeX                                         | **No**                                         | LaTeX format missing; e-TeX missing.                                                                                                                                                                  |
| `<<>>` Go template delimiters                         | Pre-processing           | N/A                                                     | N/A (handled by Go before TeX sees the source) | Unaffected by this decision.                                                                                                                                                                          |

**Score:** zero out of every package the templates currently load. This is not
"some packages missing" — it is the wrong engine class.

### Why `fontspec` is the load-bearing block

Even if every macro-only package above could somehow be coaxed onto a plain-TeX
engine (it cannot, without LaTeX2e), `fontspec` is dispositive. `fontspec`
is implemented as macros that call XeTeX / LuaTeX engine primitives:

- `\XeTeXglyph`, `\XeTeXcharglyph`, `\XeTeXfeaturename`
- OpenType font loading via the engine (not via TFM)
- Unicode codepoint handling (TeX's input is 8-bit; XeTeX is UTF-32 internally)

These are **engine features**, not macros. Adding them to `star-tex` is not a
package-loading exercise; it is rewriting the engine. The `star-tex` author's
stated roadmap is to refactor what exists, not to grow into a XeTeX
reimplementation.

### Output format mismatch

`star-tex` writes DVI natively. Its `dvi/dvipdf` package then renders DVI to
PDF as a separate step, much like `dvips` + Ghostscript. PDF-native features
the templates rely on — colour, hyperlinks, embedded PDF graphics, font
embedding of TrueType fonts — are not first-class. The current pipeline emits
PDF directly from XeTeX with all those features wired into the engine.

## Feasibility verdict

| Scenario                                                         | Verdict                               | Reason                                                                                                                                                                                                   |
| ---------------------------------------------------------------- | ------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Drop-in replacement for `xelatex` in `runXeLatex`                | **Non-starter**                       | Wrong engine class, no LaTeX2e, no fontspec, no graphicx, no hyperref.                                                                                                                                   |
| Partial replacement (e.g. simple summary pages)                  | **Non-starter for current templates** | Even the simplest template loads `fontspec` for the report font. Removing the font requirement would be a product regression.                                                                            |
| Long-term aspirational target                                    | **Not without a different upstream**  | Would need a pure-Go XeTeX or LuaTeX reimplementation. None exists.                                                                                                                                      |
| Use `star-tex` for something narrower (e.g. equations in charts) | **Possible but out of scope**         | `star-tex`'s original motivation was math rendering for `gonum/plot`. If we ever needed to render TeX equations _inside_ an SVG chart we could revisit it. The PDF report pipeline is not that use case. |

**Recommendation: NO-GO. Do not migrate. Continue investing in D-08
(precompiled `.fmt`) to shrink the vendored TeX tree.**

## What it would take to make this viable (and why we won't)

For completeness, here is what a real port would require. This is a multi-year
research programme, not a feature.

1. **Pure-Go e-TeX layer.** Add e-TeX primitives (`\protected`, `\unexpanded`,
   `\detokenize`, expanded `\numexpr`, etc.) to `star-tex`. ~30 primitives,
   each with edge cases.
2. **Pure-Go XeTeX engine.** Replace `star-tex`'s 8-bit input pipeline with
   UTF-32, add OpenType font loading via `golang.org/x/image/font/sfnt` or
   similar, implement `\XeTeXglyph` and ~50 other XeTeX primitives, integrate
   HarfBuzz-equivalent shaping (no pure-Go HarfBuzz exists).
3. **PDF backend rewrite.** Replace `dvi/dvipdf` with a direct PDF backend
   that supports object writing for `hyperref`, image embedding for
   `graphicx`, and TrueType font embedding.
4. **LaTeX2e format compilation.** Run `latex.ltx` against the new engine and
   fix incompatibilities. Validate every loaded package against the template
   set.
5. **Font asset migration.** Re-bundle Atkinson Hyperlegible in whatever
   format the new engine consumes.

Estimated effort: comparable to the original XeTeX project, which took years
of full-time work by Jonathan Kew. We are a community-traffic-monitoring
project.

## Migration phases (if the package were viable — counterfactual)

Documented for the record. **We are not executing these.**

1. **Phase 0 — Spike.** Compile a one-page hello-world template against
   `star-tex` to confirm pre-conditions. Outcome: would fail at the first
   `\documentclass` directive.
2. **Phase 1 — Backend abstraction.** Introduce an interface in
   `internal/report/` so `runXeLatex` becomes one of several backends behind
   `VELOCITY_PDF_TYPESETTER={xelatex,star-tex}`.
3. **Phase 2 — Template fork.** Maintain a parallel set of templates that
   avoid `fontspec`, `hyperref`, and `graphicx`. Quality regression accepted
   only behind a feature flag.
4. **Phase 3 — Font swap.** Generate Type 1 + TFM for Atkinson Hyperlegible.
   Visual diff against XeTeX output.
5. **Phase 4 — Image pipeline.** Convert chart SVG → EPS instead of PDF; embed
   via DVI specials; run `dvi/dvipdf` to render.
6. **Phase 5 — Acceptance.** Side-by-side golden tests: layout, page count,
   text positioning, colour fidelity. Flip default if green.
7. **Phase 6 — Removal.** Drop xelatex backend and the vendored TeX tree.

Each phase past 1 is gated on the previous one succeeding. Phase 0 fails, so
none of this proceeds.

## Risks (in the counterfactual)

| Risk                  | Severity     | Notes                                                                                                                                       |
| --------------------- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Font rendering parity | **Critical** | Type 1 conversion of Atkinson Hyperlegible will not be a pixel match.                                                                       |
| Missing packages      | **Critical** | Every package in the preamble would need a re-implementation or substitute.                                                                 |
| PDF feature loss      | **High**     | Hyperlinks in the PDF disappear without a `hyperref` equivalent.                                                                            |
| Math rendering        | **Medium**   | `amsmath` is macro-heavy; LaTeX format absence kills it.                                                                                    |
| Performance           | **Low**      | `star-tex` is single-threaded Go; XeLaTeX on a Pi 4 currently runs in ~3–5 s, so even a 2× slowdown is acceptable. Not the limiting factor. |
| Upstream maintenance  | **High**     | GitHub mirror archived; SourceHut at v0.7.0; bus factor ~1. We would own bug fixes.                                                         |
| Output fidelity       | **Critical** | Goldens will diverge for every template; visual review burden becomes ongoing.                                                              |

## Decision points (what would change our mind)

We would re-open this question if **any** of the following happens:

1. A pure-Go XeTeX or LuaTeX implementation appears with active maintenance
   and runs `fontspec` against TrueType fonts.
2. Product requirements change such that the report no longer needs custom
   fonts, hyperlinks, or embedded PDF graphics. (Unlikely; accessibility of
   the Atkinson Hyperlegible typeface is a deliberate choice.)
3. The vendored TeX tree becomes a hard blocker for distribution that D-08
   cannot resolve. (D-08 targets ~30–60 MB, which is acceptable.)

Until then, the gap is too wide to bridge with any reasonable engineering
investment.

---

## Fallback / current strategy

**Keep `xelatex` as the sole typesetter.** No env-flag transition is warranted
because there is no viable alternate backend to flag-gate.

The active workstreams that address the _real_ underlying problem (vendored
TeX size and deployment complexity) are:

- **D-08 — Precompiled `.fmt` plan.** Reduces the vendored TeX tree from
  ~800 MB to ~30–60 MB by precompiling format files and stripping unused
  packages. This is the correct lever.
- **D-09 — Distribution packaging plan.** Treats the TeX tree as an opaque
  asset. Single-binary distribution is still achievable with TeX as a sibling
  artifact in the disk image; "single Go binary" is not the only viable
  shape.

## Architecture decision record

- **Decision:** Do not adopt `star-tex.org/x/tex` as a typesetting backend.
- **Date:** 2026-04-24.
- **Status:** Accepted (NO-GO).
- **Context:** Investigated as a way to remove the vendored TeX Live tree and
  collapse the PDF pipeline into a single Go binary.
- **Alternatives considered:**
  1. Adopt `star-tex` as a drop-in replacement. **Rejected:** wrong engine
     class (plain TeX vs XeTeX); cannot run `fontspec` or LaTeX2e.
  2. Adopt `star-tex` for a subset of templates. **Rejected:** every
     template depends on the report font via `fontspec`.
  3. Continue with `xelatex` and pursue D-08. **Accepted:** addresses the
     real problem (image size) without breaking output fidelity.
  4. Wait for a pure-Go XeTeX. **Deferred indefinitely:** no such project is
     known.
- **Consequences:**
  - The Pi image continues to vendor a TeX tree (size addressed by D-08).
  - The current PDF flow continues to shell out via
    `internal/api/server_reports_generate.go` into the Python generator, which
    then invokes `xelatex`.
  - The `VELOCITY_TEX_ROOT` env contract continues to be the integration
    point for vendored TeX.
  - No Go report-pipeline migration code is introduced as a result of this
    study.

## Failure registry

For the rejected migration (documented so the failure modes are visible):

| Component                             | How it fails                                                   | Recovery                                                                |
| ------------------------------------- | -------------------------------------------------------------- | ----------------------------------------------------------------------- |
| `star-tex` engine on `\documentclass` | LaTeX format absent → fatal parse error on first template line | None. Cannot proceed.                                                   |
| `\setmainfont`                        | Undefined control sequence in plain TeX                        | Drop fontspec → report typesets in Computer Modern; product regression. |
| `\includegraphics{chart.pdf}`         | Plain TeX has no graphicx; even if loaded, no PDF embedding    | Convert charts to EPS; quality regression.                              |
| `\href` (hyperref)                    | hyperref unloadable; PDF object writing absent                 | Strip links; loss of accessibility / navigation.                        |
| Upstream package abandonment          | GitHub mirror archived; bus factor ~1                          | We become sole maintainer of a TeX engine.                              |

For the accepted plan (continue with xelatex, lean on D-08):

| Component                                     | How it fails                                   | Recovery                                                                                          |
| --------------------------------------------- | ---------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| `xelatex` missing on host                     | Build/runtime error; pipeline aborts           | `VELOCITY_TEX_ROOT` points at vendored tree; image install pre-flight check.                      |
| Font missing under vendored tree              | XeTeX falls back to default; visual regression | Detect via current PDF generator/package-parity checks; add a golden-PDF CI check as future work. |
| Vendored tree size regresses past D-08 budget | Image bloat                                    | D-08 acceptance test enforces upper bound.                                                        |
