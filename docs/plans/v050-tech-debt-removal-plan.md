# v0.5.0 Tech Debt Removal Plan

- **Status:** Active
- **Scope mode:** REDUCTION — minimum viable cleanup before v0.5.0 ships
- **Target:** Remove all remaining compatibility shims except VRLOG JSON
  (`Track.UnmarshalJSON` legacy speed-key fallback), which is deferred to
  v0.5.2 per existing plan.
- **Timeframe:** ≤ 1 week

---

## 1. Context

v0.5.0 is a coordinated breaking-change release. The bulk of the backward
compatibility shim removal is already complete (§2–§17 of the
[shim removal plan](v050-backward-compatibility-shim-removal-plan.md)).
Schema migrations 000030 and 000031 have landed. Config restructure Phase 1
and Phase 2 wiring are complete; CLI flag deprecation remains pending (A3).

This plan consolidates the **remaining** technical debt items across the
codebase, classifies each as remove-now vs retain/defer, and provides a
single checklist for the final cleanup sprint before v0.5.0 ships.

### What we are keeping

| Item                                                        | Reason                                                        | Removal target         |
| ----------------------------------------------------------- | ------------------------------------------------------------- | ---------------------- |
| VRLOG `Track.UnmarshalJSON` legacy speed-key fallback (§18) | Migration window for pre-#352 `.vrlog` files must close first | v0.5.2                 |
| `cmd/deploy` and `internal/deploy/`                         | Gated on #210 Raspberry Pi image pipeline                     | v0.7.0+                |
| Type aliases in `internal/lidar/aliases.go`                 | Architectural package-boundary choice, not a compat shim      | Retain                 |
| gRPC stream type aliases (generated)                        | Auto-generated by protoc                                      | Retain                 |
| DB legacy detection in `db.go`                              | Needed for upgrades from pre-migration databases              | Retain                 |
| Old migration files (000001–000019)                         | Immutable history — never modify applied migrations           | Retain                 |
| Config migration tool (`cmd/tools/config-migrate`)          | User migration convenience for flat → v2 config               | After migration window |

## 6. Decision Notes

1. **VRLOG speed-key fallback is explicitly retained.** The migration window
   for pre-#352 `.vrlog` files has not closed. Removing it now would silently
   lose speed data on replay. Deferred to v0.5.2 per shim removal §18.

2. **Deploy tool is externally gated (C1).** Removal is gated on four conditions
   in the retirement gate (image pipeline operational, packaging confirmed,
   migration period elapsed, zero active users). None are met.

3. **Config migration tool is retained.** Users migrating from v0.4.x need
   `make config-migrate` to convert their flat tuning JSON to v2 format.
   The tool will be removed when the migration window closes.

4. **Type aliases are architectural, not compat.** The `internal/lidar/`
   package aliases provide a stable package boundary. They are not legacy
   shims and will not be removed.

5. **Sprint scope is deliberately conservative.** Only Category A items are
   required for v0.5.0. Category B items (structural debt) are stretch goals
   that land if time permits and fall to v0.5.1 otherwise. Category C items
   (deploy tool, Python PDF) are externally gated and cannot land regardless
   of available time.
