# Split/Merge Repair Workbench

- **Source plan:** `docs/plans/lidar-visualiser-split-merge-repair-workbench-plan.md`

Non-destructive repair layer for correcting track identity errors (splits and merges) without altering underlying observations.

## Design Principle

Repairs are an **overlay**, not a mutation. The original observations remain unchanged. A repair record declares that two tracks should be treated as one (merge) or that one track should be divided (split). The effective identity resolver computes the corrected view at query time.

## Repair Types

| Type    | Meaning                                                                  |
|---------|--------------------------------------------------------------------------|
| `MERGE` | Two or more tracks are declared to represent the same physical object    |
| `SPLIT` | A single track is declared to contain observations from distinct objects |

## Suggestion Engine

Automatic suggestions generated by analysing adjacent tracks for:

- **Temporal overlap** — tracks that end/begin within a short window.
- **Centroid continuity** — last position of track A close to first position of track B.
- **Velocity continuity** — velocity vectors at boundary are consistent.
- **Box size compatibility** — bounding box dimensions are similar.
- **Class compatibility** — both tracks share the same (or compatible) classification.

## Storage

### `lidar_run_track_repairs`

| Column             | Type       | Description                    |
|--------------------|------------|--------------------------------|
| `id`               | INTEGER PK | Auto-increment                 |
| `run_id`           | TEXT       | Run identifier                 |
| `repair_type`      | TEXT       | MERGE or SPLIT                 |
| `source_track_ids` | TEXT       | JSON array of source track IDs |
| `target_track_id`  | TEXT       | Resulting effective track ID   |
| `state`            | TEXT       | SUGGESTED, ACCEPTED, REVERTED  |
| `actor`            | TEXT       | User or system                 |
| `reason`           | TEXT       | Free-text justification        |
| `created_at`       | TEXT       | ISO 8601                       |
| `updated_at`       | TEXT       | ISO 8601                       |

## Effective Identity Resolver

A query-time function that maps raw track IDs to effective track IDs by walking the accepted repair chain. All downstream consumers (quality scoring, review queue, audit, export) use effective identity.

## Side Effects on Accept

When a repair is accepted:

1. Apply repair → update `lidar_run_track_repairs.state` to ACCEPTED
2. Emit `REPAIR_APPLIED` track event
3. Recompute quality score for affected effective track
4. Refresh review queue priority for affected track
5. Append entry to `lidar_qc_audit_log`
